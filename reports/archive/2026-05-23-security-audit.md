## Findings (sorted by severity, Critical first)

| #  | Severity | Category   | File:Line | Finding | Recommendation |
|----|----------|------------|-----------|---------|----------------|
| 1  | CRITICAL | Auth       | crates/copypaste-relay/src/state.rs:107-114 | Bearer token derivation `SHA256(public_key)[0..32 hex]` is **fully deterministic from a publicly-shared value**. The `public_key_b64` is sent in plaintext on `POST /devices` and exposed via `GET /devices/:id` and `GET /devices` (list). Any attacker who reads the public key (or who registers a device that later queries the listing) can compute the bearer token offline and impersonate the device — push/pull/delete arbitrary items. Equivalent to "no auth". | Replace deterministic derivation with a cryptographically random secret token (e.g. 32 bytes from `OsRng` base64-encoded). Never expose the raw public key on read endpoints, or unlink auth token from public key entirely. Persist the random token only in the device record. |
| 2  | CRITICAL | Auth       | crates/copypaste-daemon/src/ipc.rs:413-433 | `get_own_fingerprint` returns a **fake fingerprint** computed from `hostname` + `std::process::id()` hashed with the non-crypto `DefaultHasher` (SipHash with random seed → not deterministic across processes, and `process::id()` is enumerable). UI surfaces this as the device identity for pairing, but it has zero relation to the real TLS cert / X25519 keypair used by the p2p crate. A user who "verifies" this fingerprint out-of-band has verified nothing, and pairing decisions made on it provide **no security**. Combined with finding #4 (p2p uses an entirely separate identity from peers.json), this means pairing UI is theatre. | Either (a) return the real cert fingerprint from `keychain::load_or_create()` / the p2p `SelfSignedCert::fingerprint()`, or (b) explicitly mark this IPC method as `unimplemented` and disable the UI pairing flow until Phase 5 wires it up. Do NOT ship a "fingerprint" that does not authenticate anything. |
| 3  | HIGH     | Auth       | crates/copypaste-daemon/src/cloud.rs:107-122 | `SUPABASE_PASSWORD` is read from environment and POSTed to `/auth/v1/token?grant_type=password`. On sign-in **failure** the daemon silently falls back to the `anon_key` as bearer (line 117-119, log only). This means a typo in password (or active MITM stripping TLS verification on a misconfigured system) **does not stop sync** — it sends data with anon privileges instead, which may be silently allowed by RLS if the Supabase project is misconfigured. The failure mode is invisible to the user. | On sign-in failure, abort cloud sync entirely (do not start push/poll loops). Surface the error via IPC so UI can show "auth failed". Never silently downgrade auth scope. |
| 4  | HIGH     | Auth       | crates/copypaste-daemon/src/ipc.rs:425-466 | Peer pairing stores `peers.json` in plaintext at `dirs::config_dir()/copypaste/peers.json` with **no `set_permissions(0o600)`** call. File is created with the process umask (typically 0o644 → world-readable on multi-user systems). On macOS shared machines or Linux servers, other local users can read the paired-peers list and modify it (any user-writable parent dir → can drop a peers.json containing attacker's fingerprint). | After `std::fs::write`, call `std::os::unix::fs::PermissionsExt::set_permissions(path, Permissions::from_mode(0o600))`. Same fix for `config.json`. Verify the parent dir is also 0o700. |
| 5  | HIGH     | IPC        | crates/copypaste-daemon/src/ipc.rs:122-125; crates/copypaste-daemon/src/daemon.rs:48-58 | The Unix domain socket at `~/Library/Application Support/CopyPaste/daemon.sock` is bound with `UnixListener::bind` and **no `set_permissions(0o600)`** afterwards. Default mode honors umask (0o755 on macOS → world-connectable). Any local process under any UID can connect to the daemon and issue every IPC command (`delete_all`, `pair_peer`, `set_private_mode`, `paste`, etc.) — full data exfiltration + tamper. | Immediately after `UnixListener::bind`, call `std::fs::set_permissions(socket_path, std::fs::Permissions::from_mode(0o600))`. Also enforce that the parent dir is 0o700 owned by the user. Consider per-connection getsockopt(SO_PEERCRED) check that peer uid == own uid as defense-in-depth. |
| 6  | HIGH     | IPC        | crates/copypaste-daemon/src/ipc.rs:145-152 | `BufReader::lines()` reads a line per IPC request with **no size cap**. A malicious local client (or a runaway UI) can send a multi-GB single line, causing the daemon to allocate unbounded memory and OOM-kill itself (denial of service). | Use `read_until(b'\n', &mut buf)` with a hard cap (e.g. 16 MiB), reject and close on overflow. Same for `lines()` in `copypaste-ui/src/ipc_client.rs` if it reads daemon responses. |
| 7  | HIGH     | Deps       | Cargo.lock — image 0.24.9 + tiff 0.9.1 | Clipboard image bytes are read directly from NSPasteboard (`pb.dataForType("public.tiff")`, `clipboard.rs:80-83`) and decoded by the `image` crate. Both `image 0.24` and `tiff 0.9` are EOL branches with known unbounded-allocation issues on malformed TIFF (parent fixed in `image 0.25` / `tiff 0.10`). Attacker who can write arbitrary bytes to your pasteboard (any local app, or a malicious clipboard manager) triggers OOM in the daemon. | Upgrade workspace to `image = "0.25"` and `tiff = "0.10"`. If MSRV blocks upgrade, gate TIFF decoding behind a size pre-check (already done at 25 MB) **and** wrap decode in `catch_unwind` + thread limit. |
| 8  | HIGH     | Deps       | Cargo.lock — rusqlite 0.31.0 / libsqlite3-sys 0.28.0 | Both crates are >1 year behind. Newer rusqlite (0.32+) carries SQLite fixes and the SQLCipher feature you use (`bundled-sqlcipher`) ships an older SQLCipher build. SQLCipher CVEs in the bundled tarball are not tracked by `cargo audit` — must be verified manually. | Upgrade to `rusqlite = "0.32"` (or current) and audit the underlying SQLCipher version (`PRAGMA cipher_version`). Document the bundled SQLCipher version in CHANGELOG so users know what they're getting. |
| 9  | MEDIUM   | Crypto     | crates/copypaste-core/src/crypto/keys.rs:41-48 | `derive_enc_key()` HKDF uses an empty salt (`Hkdf::<Sha256>::new(None, ...)`). Per RFC 5869 §3.1, salt should be a non-secret random per-context value; omitting it weakens collision resistance when the same IKM (ECDH output) is reused across many info strings. Low practical impact (info string already includes sender_id/recipient_id) but defense-in-depth gap. | Add a per-conversation salt: `Hkdf::new(Some(b"copypaste-v1-salt"), &raw_secret)` or, better, derive salt from `hash(sender_pub || recipient_pub)`. |
| 10 | MEDIUM   | Crypto     | crates/copypaste-core/src/crypto/encrypt.rs:25 | `cipher.encrypt(...).expect("XChaCha20-Poly1305 encryption cannot fail for valid inputs")`. | 11 | MEDIUM   | Crypto     | crates/copypaste-daemon/src/daemon.rs:401-403, 421 | When Keychain is unavailable (e.g. Keychain locked, sandbox denial), the daemon silently falls back to `DeviceKeypair::generate().local_enc_key()` — an **ephemeral key**. On next restart, every previously-encrypted item in SQLCipher is unreadable (decryption returns "wrong key"). User sees an empty history with no warning; no error path informs them their data is lost. | If Keychain access fails, refuse to open the existing encrypted DB. | 12 | MEDIUM   | Crypto     | crates/copypaste-core/src/storage/db.rs:81-105 | Plaintext-to-encrypted migration path: if SQLCipher open fails with `SQLITE_NOTADB`, the code falls back to opening the file **unencrypted** and migrating. A user who rotated keys (e.g. wiped Keychain entry but DB file remains) will trigger the wrong-key path → unencrypted probe → fail → return error — fine. But the same probe leaks information: an attacker with read access to the file (e.g. Time Machine snapshot) can call `Database::open` with any wrong key and learn whether the file is plaintext or encrypted via the error message timing. Low risk (already an SQLCipher property) but the migration path is the noisy side-channel. | Add a one-time migration marker file (e.g. `clipboard.db.sqlcipher`) instead of probing. | 14 | MEDIUM   | Network    | crates/copypaste-relay/src/routes/devices.rs:43-50; crates/copypaste-relay/src/state.rs:108 | No proof-of-possession on `POST /devices`. ## Blocker for alpha release?

**YES** — findings #1, #2, and #5 must be resolved before any public alpha.

Rationale:
- **#1 (deterministic relay bearer token)** is full account takeover for anyone who sees the public key. The public key is broadcast on every device list endpoint. This is not a theoretical issue.
- **#2 (fake fingerprint in IPC)** makes the pairing UI cryptographically meaningless. Shipping a "verify this fingerprint" UI that verifies nothing is worse than no UI — users will trust it.
- **#5 (world-connectable IPC socket)** lets any local process own the daemon. On a single-user Mac this is "only" cross-app exposure, but a malicious app that lands on the system has full clipboard access without any prompt.

Findings #3, #4, #6, #7, #8 are all **must-fix before release**, but each has a short, well-scoped patch. None require architectural change.

Findings #15-#16 (P2P stubs / mDNS spoofing) are acceptable in alpha **only if** the P2P pairing UI is explicitly labelled "preview / not secure" and the corresponding IPC methods return `not_implemented`. Currently they return success with fake data, which is dangerous.

The cryptographic primitives themselves (XChaCha20-Poly1305 with chunk AAD binding file_id+chunk_index+total_chunks+is_final, HKDF-SHA256 key derivation, X25519 ECDH, fingerprint-pinned mTLS with custom rustls verifier) are well-designed and correctly implemented. Most of the audit findings are at the **integration / glue layer**, not in the cryptography core.
