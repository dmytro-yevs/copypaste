## CRITICAL — sync

- [~] **B2 — Sync very slow (30s–1min+ latency).** A clip copied on device A takes 30s to
  over a minute to reach device B. Must be near-realtime. *Status: opus investigator tracing
  every poll/sleep on the end-to-end path (capture cadence, push trigger, relay poll interval,
  P2P push-vs-pull). Prime suspect: a fixed 30–60s relay/sync poll. → fix plan incoming.* **P0**
- [~] **B3 — Images may not sync macOS→Android at all.** User can't confirm images arrive on

  *Status: opus investigator checking the Android receive/decrypt/store/render path.* **P0**

## CRITICAL — sync / connectivity

- [~] **HB-7 — Sync delivers but Android STORES 0. (THE "sync doesn't work".)** P0. Android Pair
  status: "received 23 item(s), **stored 0**, sent 0". mTLS catch-up transfers items but Android
  persists none, and pushes nothing back. *Diagnosing (a15c84e1): LWW dedup vs per-item decrypt-fail
  vs store-never-called vs counter mislabel.*
- [~] **HB-2 — macOS ↔ Android pairing/connect dead (bidirectional).** P0. Mac→Android discovery Pair:
  `Connection refused (os error 61)`; Android→Mac: `tls handshake timed out after 10s`. ROOT (confirmed
  ×2 agents): Android's discovery BootstrapResponder (bport 47654) + mDNS advert are bound to the
  DevicesActivity SCREEN lifecycle (DisposableEffect), not the always-on ClipboardService FGS; macOS
  responder is process-lifetime. Stale mDNS on Mac (no TTL aging, discovery.rs:527). FIX: move
  startDiscovery/stopDiscovery into ClipboardService FGS; add macOS discovery last_seen TTL.
- [~] **HB-steady — Steady-state sync after pairing.** P0. "First/Last sync —" on Mac vs "Last sync 3m"
  on Android = asymmetric; need to confirm ongoing Mac↔Android sync runs at all (sync_addr reachability +
  Android pull-gating). *Folded into a15c84e1.*

## Device info (BUG 1 — three gaps, RECURRING)

- [~] **HB-1 — Device metadata not gathered/exchanged/shown.** P0. (i) Android "This Device" shows only
  Device ID + fingerprint (no model/OS/version/local IP/public IP) — Android never GATHERS its own info,
  though Build.MODEL/Build.VERSION.RELEASE/BuildConfig.VERSION_NAME are trivially available; (ii) Android
  sends `PeerMeta::default()` (lib.rs:629/1567/1707) → Mac shows the Android peer as bare "Device b7:d2:3b";
  (iii) `BootstrapResult` carries no peer meta → Android shows the Mac peer as just "CopyPaste" + fingerprint
  + sync_addr. PeerMeta struct + macOS storage/render already complete (no struct change). FIX: Android gather
  + display own card; populate PeerMeta on send (FFI param add → ABI bump + re-pair); extend BootstrapResult
  to carry Mac meta; Android render peer meta; macOS UI add peer **Public IP** row (DevicesView.tsx:202).
