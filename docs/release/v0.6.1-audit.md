## EXECUTIVE SUMMARY (honest)
The **Rust core + macOS daemon are solid**; **security is strong (0 critical)**. The v0.6.0 "three
transports + full parity" story is **substantially aspirational on the cross-device / Android side**:
- **Relay transport is DEAD in shipped builds** — the daemon is never compiled with `relay-sync` (CI builds
  cloud-sync only). All relay daemon code is correct but `#[cfg]`-d out of every artifact.
- **QR full-provisioning of Supabase does NOT work** — the QR carries no account credential/bearer; RLS
  needs `authenticated`, so a QR-provisioned phone silently fails every cloud op (and falsely shows "configured").
- **Android client is broadly incomplete vs macOS**: can't originate files, corrupts cloud-received files,
  never uploads files/images to cloud, sends a URI placeholder instead of image bytes, drops all sensitive
  items, has no device-info, uncapped image memory, P2P discovery dies with the screen, no FLAG_SECURE on history.
- **P2P plumbing/crypto is correct and symmetric**, but the product is broken end-to-end (HB-2/7 + Mac→Android push).

## 🔴 P0 — core-broken / data-loss (release-blocking for "it works")
- **AB-1 Relay never compiled into the daemon.** `release.yml:104` + `build-macos.sh:24` build `cloud-sync` only;
  no `default = ["relay-sync"]`. Entire daemon relay path dead in production. FIX: add `relay-sync` to CI + local build.
- **AB-2 Supabase QR provisioning credential gap.** `SyncProvisioning` (bootstrap.rs:312, lib.rs:514) carries
  url+anon+derived_key but NO email/password/bearer/refresh-token. RLS is `authenticated`-only → QR-provisioned
  phone uses anon bearer → every insert/select silently rejected. `isSupabaseConfigured` ignores credentials →
  false "configured". FIX: carry a Supabase refresh-token in SyncProvisioning over the PAKE tunnel + apply it;
  gate "configured" on a real session.
- **AB-3 Android cloud file RECEIVE corrupts data.** Android has no `decode_cloud_file_payload` mirror; stores the
  header-prefixed plaintext as the file body + name/mime null (FgsSyncLoop.kt:364, SupabaseRealtimeClient.kt:539,
  SyncManager.kt:359). Every Mac→Android file over Supabase/relay = broken bytes. FIX: port sync_common.rs header.
- **AB-4 Android never UPLOADS files/images to cloud.** `notifySyncManager` hardcodes `contentType="text"`
  (ClipboardService.kt:919,944). Image/file captures never push to Supabase/relay. FIX: route by type + encode header.
- **AB-5 Android→macOS IMAGE send broken.** `localItemsForSync` has a file branch but NO image branch
  (ClipboardRepository.kt:1228) → sends the `content://` URI placeholder string, not bytes (ClipboardService.kt:783,805).
- **AB-6 Sensitive sync incoherence + threshold mismatch.** macOS stores sensitive (masked + TTL ≥0.70);
  Android DROPS sensitive at capture AND at sync-in (ClipboardRepository.kt:556,622) and flags on ANY match
  (lib.rs:133). → macOS sensitive items never reach Android; mildly-sensitive items vanish. FIX: align policy
  (store+mask+TTL) + threshold parity.
- **AB-7 No FLAG_SECURE on history/clips.** Only PairActivity is secure (PairActivity.kt:144). Full clipboard
  history (incl. UI-masked items) is screenshot-able + shows in Android recents. FIX: FLAG_SECURE on MainActivity.
- **AB-8 Android image memory uncapped + eager.** `getItems(200)` attaches decoded bytes for EVERY image up front
  (ClipboardRepository.kt:110-126); no LRU/lazy/virtualized cap (unlike macOS 24MiB). Root of Android jank + OOM risk.
- **HB-2 P2P discovery dies with the screen.** Android responder+mDNS bound to DevicesActivity DisposableEffect,
  not the FGS → Mac→Android ECONNREFUSED, Android→Mac TLS-timeout. FIX: host discovery in ClipboardService FGS.
- **HB-7 "received N stored 0".** Silent per-item drop (decrypt-fail/legacy) + counter shows raw frames; PairActivity
  force-decodes all as text. + Mac→Android steady push dead (ephemeral listener port, stale persisted addr).
- **HB-1 Device-info 3 gaps.** Android gathers none of its own info; sends PeerMeta::default(); BootstrapResult
  drops Mac's meta. Both cards bare. (ABI bump + re-pair.)
