  bootstrap.rs:488 (responder) / :651 (initiator), AFTER PAKE finish AND channel binding.
  Transcript+channel bound, identical on honest endpoints, different per relay leg.

**KEY ARCHITECTURAL BUG:** IPC `ServerState` already has `discovery: Option<Arc<DiscoveryService>>`
(ipc.rs:879) + `with_p2p_discovery` setter (ipc.rs:1046), but `start_p2p` constructs its OWN
separate `DiscoveryService` (p2p.rs:283) that is NOT handed to IPC. To surface a live
discovered list the daemon must share ONE `Arc<DiscoveryService>` between `start_p2p` and the
IPC server. `copypaste-daemon/src/daemon.rs` (~611) wires `start_p2p`.

## SAS design rationale
QR path: password = high-entropy `PairingToken` only the two devices know → PAKE alone
authenticates. Discovery path: NO shared secret → bootstrap runs with an **ephemeral random
password the initiator sends in-clear inside bootstrap TLS**, and authentication is provided
ENTIRELY by the human SAS compare. SAS MUST derive from `bound_key` (post-PAKE,
post-channel-binding) so a MITM substituting its own password per leg yields a different
`bound_key` per leg → different SAS → user sees mismatch (Bluetooth numeric-comparison /
Magic-Wormhole-verifier pattern). 6 decimal digits ≈ 20 bits; single-shot (ephemeral keys,
abort on mismatch) → 1-in-10^6 online forgery.

## Phase 2 — Daemon IPC + pairing state machine
New module `copypaste-daemon/src/pairing_sm.rs`: `Idle → Initiating → AwaitingSas{sas,role,
expires_at} → Confirmed|Rejected|Aborted|TimedOut`. `Arc<Mutex<PairingState>>` in ServerState.
One active pairing (reject concurrent, `ERR_CODE_RATE_LIMITED`). Timeout: `PAKE_EXCHANGE_TIMEOUT`
(30s) + `SAS_CONFIRM_TIMEOUT=60s`. `confirm` callback transitions SM → `AwaitingSas` then
awaits a `oneshot` fired by `pair_confirm_sas`.
- **2.1** `pairing_sm.rs`: states, transitions, `pending_sas: Mutex<Option<(String, oneshot::Sender<bool>)>>`.
  Register `mod pairing_sm;` (where `mod peers;` is).
- **2.2** `methods.rs`: `METHOD_PAIR_WITH_DISCOVERED`, `_PAIR_GET_SAS`, `_PAIR_CONFIRM_SAS`, `_PAIR_ABORT`.
- **2.3** `ipc.rs` `"pair_with_discovered"` near `pair_peer` (3260): resolve via
  `discovery.resolve_peer` (242), gen ephemeral password (`PairingToken::generate()`), INITIATOR
  calls `run_initiator_with_confirm`. Success → `persist_paired_peer` (1294) + `rotate_peer` (1499).
- **2.4** `ipc.rs` `"pair_get_sas"` (poll → `{state,sas?,role?}`), `"pair_confirm_sas"`
  (`{accept}` fires oneshot), `"pair_abort"` (abort handshake task, SM→Idle).
- **2.5** Standing discovery-pairing responder task in `start_p2p` (p2p.rs ~337): binds one
  ephemeral bootstrap listener (port in new TXT `bport`), accepts, runs `run_with_confirm`,
  routes SAS through SM/oneshot. Password transmitted in-band (SAS is the authenticator).
- **2.6** `discovery.rs`: TXT key `bport` (35-41, advertise 263-267, `PeerInfo` 55-65,
  `peer_from_resolved` 428). Bump `PROTOCOL_VERSION` → "2"; accept v1 (no bport, pair disabled) + v2.
- Verify: `cargo clippy -p copypaste-daemon -p copypaste-p2p -p copypaste-ipc`

## Open questions / risks (ranked) + v0.6 scope
1. **(High) Responder consent with no foreground UI** — needs OS notification ("Pair with
   <name>? Code: 123456") deep-linking to SAS dialog; auto-reject after `SAS_CONFIRM_TIMEOUT`.
   macOS Tauri notification + Android `NotificationHelper.kt` exist.
2. **(High) Android in-process mDNS** — daemon discovery is desktop-only; use `NsdManager` on
   Android (least risk) not Rust-discovery-over-UniFFI.
3. **(Med) Single active pairing** — SM rejects concurrent (v0.6 simplicity).
4. **(Med) `bport` TXT + PROTOCOL_VERSION "2"** — v1 peers: show in list, disable "Pair" button.
5. **(Low) SAS length/encoding** — 6 digits over words (cross-locale, parity w/ BT numeric).

**Recommended v0.6 scope:** Phases 0-3 fully + Phase 5 docs/tests; Android (Phase 4) as
discovered-list + SAS via NsdManager + two-call `pair_begin`/`pair_confirm`, ZXing QR fallback.
Defer responder-notification polish (risk #1) to fast-follow, gating discovery pairing behind a
setting until both-sides-consent UX is verified on hardware.
