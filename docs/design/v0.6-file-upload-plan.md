## Locked decisions
- Content-type = `"file"`, reuse `content` (chunked blob) + `blob_ref` JSON
  `{"filename","mime","original_size","chunk_count","file_id":[u8;16]}`. **No new columns**
  (avoids touching the 6 SELECTs + mapper). `content_nonce = None`. **No decode/re-encode** —
  files are chunked RAW (not through the `image` crate).
- Size cap: new `MAX_FILE_BYTES` + config `max_file_size_bytes`. **RISK: relay caps `"file"` at
  10 MiB (`quota.rs:43`) but local default 100 MiB → a 100MiB file syncs P2P but relay-413s.
  Recommend default 10 MiB to match relay, OR bump relay.** Files count toward byte cap (no
  prune change needed).
- Copy-back: macOS writes temp file + `public.file-url`; Android FileProvider content URI.
  Text/image branches UNTOUCHED — add `else if content_type=="file"`. Regression-test guard.

## Risks (ranked)
1. HIGH copy-back fidelity + temp-file lifecycle (macOS & Android) — least-tested, manual checklist. → v0.7.
2. HIGH 17-col SELECT/mapper fan-out — mitigated by no-new-columns; T2.5 is the guard. RESIST adding columns.
3. MED size-cap vs relay 413 (T7.3). 4. MED Android MIME/filename from content URIs (SAF quirks).
5. MED prune_to_cap never evicts newest unpinned (a >quota file survives one cycle). 6. LOW cloud. 7. LOW search (no FTS for files; optionally index filename).
