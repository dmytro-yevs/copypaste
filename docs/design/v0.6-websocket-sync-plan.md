# v0.6 — WebSocket-everywhere sync (authoritative plan)

> 2026-06-03. Move sync to push everywhere; eliminate polling as the primary receive path so
> clips propagate in ~1s. Base: `v0.6-integration`.

## Verified state
- macOS daemon ALREADY has Supabase Realtime WS: `cloud.rs:1868` `ws_ingest_loop`, fallback poll
  120s (WS up) / 10s (WS down) `cloud.rs:80/85`. Protocol impl: `copypaste-supabase/realtime.rs`+`protocol.rs`.
- Android has NO WS — polls (`FgsSyncLoop.kt`: 60s/idle, now interim 3s stopgap merged; `SupabasePollWorker` 15min). `SupabaseClient.kt` is REST-only (HttpURLConnection).
- P2P already push over persistent mTLS — no change.
- Relay incoming path is DEAD CODE on Android (`SyncManager.kt:201` @Deprecated/throws) and absent on
  the daemon → relay WS/SSE is CONDITIONAL (only if relay sync is reactivated).

## Supabase Realtime wire protocol (Android must mirror byte-for-byte)
- WS URL: `wss://{project}.supabase.co/realtime/v1/websocket?apikey={ANON}&vsn=1.0.0` (realtime.rs:301).
- Frame = 5-elem JSON array `[join_ref, ref, topic, event, payload]` (protocol.rs:14).
- Topic `realtime:clipboard_items`. Join `phx_join` with `{config:{access_token:<USER_JWT>,
  postgres_changes:[{event:"*",schema:"public",table:"clipboard_items",filter:"user_id=eq.<UUID>"}]}}`.
- Heartbeat every 30s: `[null,"<ref>","phoenix","heartbeat",{}]`. Handle phx_reply(ok)=joined,
  postgres_changes=data (row at payload.data.record), phx_error/phx_close=reconnect. Leave=phx_leave.
- INSERT row fields: id,item_id,payload_ct(bytea hex, decode via SupabaseClient.decodePayloadCt:94),
  content_type,lamport_ts,wall_time,device_id,expires_at,app_bundle_id.

## Phase 1 — Android Supabase Realtime WS (highest impact, ships first)
- P1.1 add OkHttp dep (`com.squareup.okhttp3:okhttp:4.12.0`) to android/app/build.gradle.kts; watch JNA @aar dup-class.
- P1.2 new `SupabaseRealtimeClient.kt` (OkHttp WebSocket + WebSocketListener) mirroring realtime.rs:
  join/heartbeat/reconnect(1s→60s jitter); reuse SupabaseClient.decodePayloadCt/decryptRow (no dup crypto).
- P1.3 factor `FgsSyncLoop.poll()` per-row body (:239-291) into shared `ingestRow(row)` used by WS + poll
  (dedup via storeItemWithLww by item_id; advance the same (wall_time,id) cursor only on forward rows;
  observe lamport :370).
- P1.4 FGS owns WS lifetime (start :130 / stop :195 / onDestroy). Demote poll to catch-up: 120s while
  wsConnected, 60s down, 300s idle; immediate one-shot poll on each (re)connect.
- P1.5 auth: apikey=anonKey; access_token=user JWT via `SyncManager.cachedOrFreshBearer`; capture GoTrue
  `user.id` for the filter (extend signIn or decode JWT sub); refresh token on reconnect.
- P1.6 revert the 3s stopgap once WS lands; rewrite FgsSyncLoop KDoc (currently argues against WS).
- **ONLY ONE Android agent may touch FgsSyncLoop at a time** — serialize vs image-sync Android work.

## Phase 2 — relay SSE (CONDITIONAL — only if relay sync reactivated)
- SSE over Axum (`Sse<impl Stream>`): per-device `Notify`/`broadcast` in relay state.rs fired from
  `push_item_decoded`; `subscribe` SSE handler in routes/items.rs (auth via BearerToken; emit new PullItems
  from `since` cursor; SSE id = item id for Last-Event-ID; 25s keepalive comments). Android RelayClient.subscribe
  (HttpURLConnection line-reader or OkHttp EventSource); re-enable SyncManager relay ingest. 5-min catch-up backstop.

## Phase 3 — macOS daemon WS hardening
- P3.1 set `ws_connected=true` only after phx_reply(ok) join-confirmed (add a joined signal in
  realtime.rs dispatch_event:817), not on socket-open. P3.2 lower POLL_INTERVAL_WS_CONNECTED 120s→60-90s.

## Cross-cutting
- 30s heartbeat both platforms; exp backoff 1s→60s jitter; WS+poll share one (wall_time,id) cursor +
  item_id LWW dedup so a WS gap is healed by catch-up with no dupes; lamport advanced on every ingest.
- Doze: FGS keeps WS alive through light Doze; deep Doze drops it → 15min WorkManager backstop + optional
  battery-opt exemption (permissioned). START_STICKY + BootReceiver restart.

## Risks: Doze/OEM killers (WorkManager backstop), SSE proxies (keepalive+long-poll fallback), Supabase
## Realtime connection quotas (1 WS/device, fine for personal multi-device), JWT expiry (reconnect w/ fresh token), OkHttp/Okio classpath dup.

## Order: P1 (unconditional, highest impact) → P3 → P2 (conditional).
