# v0.6 — Android LAN/SAS discovery pairing (gap #14, Phase 4) — implementation plan

> 2026-06-03 (investigator a054fba9, read-only). Brings discovery + 6-digit SAS pairing to Android at
> parity with macOS DevicesView. Android has ONLY QR pairing today. ABI bump (combine with mTLS listener).

## Primitives (copypaste-p2p)
- `run_initiator_with_confirm(addr, cert_der, key_der, password, sync_addr, own_meta, own_provisioning, confirm: FnOnce(&str)->Future<bool>) -> BootstrapPairing` (bootstrap.rs:1085); SAS derived at :1202 via `derive_sas(&bound_key)`, `confirm(&sas).await` OUTSIDE the PAKE deadline. Both sides exchange frame-10a SAS_ACCEPT; success only if both accept; reject/error → key zeroizes, persist nothing.
- `BootstrapResponder::run_with_confirm` (bootstrap.rs:749) — responder mirror (makes Android discoverable/pairable inbound).
- `derive_sas` (re-export bootstrap.rs:118) — 6-digit from post-channel-binding bound_key; MitM → divergent SAS.
- `BootstrapPairing` (bootstrap.rs:444): peer_fingerprint, peer_sync_addr, session_key, sas, peer_* meta, peer_provisioning.
- Discovery (discovery.rs): `DiscoveryService::new`/`register_with_bport(port,id,name,bport)` (:206, v2 TXT `bport`)/`start`(:243)/`peers()->[PeerInfo]`(:324)/`resolve_peer`(:346). `PeerInfo{device_id,device_name,ip_addrs,port,bport:Option<u16>}` (bport None = v1 peer, Pair disabled).

## macOS reference: `pairing_sm.rs` PairingCoordinator (Idle→Initiating→AwaitingSas{sas,role,expires}→Confirmed|Rejected|Aborted|TimedOut, oneshot<bool>, try_begin single-active, SAS_CONFIRM_TIMEOUT 60s) + IPC verbs list_discovered/pair_with_discovered/pair_get_sas/pair_confirm_sas/pair_abort + UI SasPairingModal (DevicesView.tsx:276, poll pair_get_sas ~500ms, Match/Doesn't-match, terminal states).

## FFI shape — POLLED STATE MACHINE (option b), NOT callback-interface
(UniFFI 0.27 callbacks invoked from inside `runtime().block_on` while the JNI thread is parked are deadlock-prone; the polled SM is a near-1:1 port of the proven macOS path.)
New `crates/copypaste-android/src/pairing.rs`: port `PairingCoordinator` (pure std::sync::Mutex + tokio oneshot, zero daemon deps) into a process-global `OnceLock<AndroidPairing>` + a JoinHandle/abort slot for the in-flight initiator task. Extend the `Confirmed` variant to hold the FFI-safe BootstrapPairing outputs (Kotlin persists, not Rust).
New UDL: `dictionary DiscoveredPeer{device_id,device_name,ip_addrs:[string],port:u16,bport:u16?,paired:bool}`; `start_discovery(device_id,device_name,sync_port,bport,cert_der,key_der)` (idempotent; also binds a standing BootstrapResponder on bport via run_with_confirm wired to the SAME coordinator with Responder role → makes Android pairable FROM macOS); `stop_discovery()`; `list_discovered(paired_fingerprints:[string])->[DiscoveredPeer]`; `pair_with_discovered(device_id,cert_der,key_der,sync_addr,local_provisioning:SyncProvisioning?)` (resolve bport+IPv4-first addr, try_begin(Initiator), ephemeral `PairingToken::generate().to_pake_password()`, SPAWN run_initiator_with_confirm on runtime — do NOT block_on — confirm closure does enter_awaiting_sas + timeout(60s,rx)); `pair_get_sas()->PairStatus`; `pair_confirm_sas(accept)`; `pair_abort()`; `pair_reset()` (Kotlin-driven on modal close, since terminal state must survive until polled). `PairStatus{state, sas?, role?, peer_fingerprint?, peer_sync_addr?, session_key?:[u8] SECRET, peer_provisioning?}` — peer_* populated only on `confirmed`.

## Kotlin (DevicesActivity): "Discovered on network" section (poll list_discovered ~2s, Pair disabled when bport==null), SAS modal (port SasPairingModal: poll pair_get_sas ~500ms, show 6-digit SAS, Match=pairConfirmSas(true)/Doesn't-match=false, terminal confirmed/rejected/aborted/timed_out/ended; close-before-terminal → pairAbort). On `confirmed`: convert session_key List<UByte>→ByteArray, settings.wrapSessionKey, settings.upsertPeer, apply peer_provisioning fill-missing (copy PairActivity.kt:383-426), zero the key, pairReset. start_discovery on screen open (P2P enabled) with DeviceKeyStore cert + fixed bport; stop_discovery on dispose.

## Security: SAS from post-channel-binding bound_key; both sides SAS_ACCEPT before key trusted; reject/abort/timeout zeroizes (persist nothing); purely additive (QR + pinning untouched); ephemeral PAKE password safe ONLY because SAS authenticates; Kotlin zeroes session_key after KEK-wrap.

## ABI ordering — COMBINE with the mTLS-listener FFI ([[v0.6-android-mtls-listener-plan]]) under ONE ABI bump.
Both are android-crate FFI + functionally coupled (responder/listener = inbound). Single android-crate FFI writer.
On the integration branch, chaining 10→11 (listener) then 11→12 (SAS) is acceptable — only the FINAL released ABI matters; intermediate bumps are invisible to users. Either combine into one agent or chain two; do NOT run two android-FFI agents concurrently. Order each: Rust/FFI → version.rs bump → regen (`scripts/generate-android-bindings.sh`) → Kotlin (+ bump CopypasteBindings APP_ABI_VERSION to match). Regen needs ANDROID_NDK_HOME + --no-configuration-cache for the fresh .so (CI/hardware for assembleDebug).
