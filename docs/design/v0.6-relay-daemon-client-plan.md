# v0.6 — Relay-as-database sync path (#2 of 3) — plan + the gating blocker

> 2026-06-03 (investigator a8194662, read-only). The relay SERVER + Android SSE RECEIVER exist and are
> consistent. The path is dead because: no daemon producer, Android producer uses a legacy/buggy format,
> AND — critically — the server has no cross-device delivery model.

## ⛔ GATING BLOCKER (needs a relay SERVER change — cannot be done client-only)
`verify_token(device_id, token)` (routes/items.rs:105) binds EVERY read+write to the caller's OWN device token.
There is no account/sync-key dimension. So one device cannot POST into, or read, another device's inbox →
**no item can travel between devices over the relay today.** The Android SSE receiver only ever reads its own inbox, which nothing else can write to.
- Smallest fix (option A, recommended): **shared-account inbox** — derive ONE `device_id` (UUID via HKDF of the sync key) that all account devices register; allow CO-REGISTRATION of the same id from different callers, each issued an INDEPENDENT token (drop the 409 in routes/devices.rs / add `POST /devices/{id}/tokens`). All devices then push to + subscribe to that one inbox. Relay still sees only one opaque inbox per account.
- (Per-device fan-out is impossible: a producer never holds a peer's token.)
- SECOND server gap: relay store is **in-memory** (store.rs) — "relay-as-database" per the user requirement needs a PERSISTENCE layer (sqlite/disk) so it survives restart.

## Server contract (verified, no change needed beyond the above)
- Register `POST /devices` {device_id: UUID, device_name: 1-64, public_key_b64: 32 bytes} → 201 {device_id, auth_token(32hex), expires_at(1yr)}. (Android producer bug: checks 200 not 201, reads `token` not `auth_token`, omits device_name — fix in RelayClient.kt:43-46.)
- Push `POST /devices/{id}/items` Bearer {content_type: text|image|file, content_b64, wall_time:u64 millis} → 201 {id}. Tier caps: text 8 MiB (post Nit-A), image/file 10 MiB.
- Pull `GET /devices/{id}/items?since=&since_id=&limit=` → [{id,content_type,content_b64,wall_time}] asc past cursor, at-least-once.
- SSE `GET /devices/{id}/subscribe?since=&since_id=` → event:item/id/data{...}, 25s keepalive.
- TTL evict at inserted_at+sync_ttl_secs; device record reclaimed after 30d empty inbox.

## Envelope (producer MUST emit; Android receiver already decodes it)
`content_b64` = base64( JSON `{item_id, lamport_ts, ct_b64}` ), where `ct_b64` = base64( `encrypt_for_cloud(sync_key, item_id, wrapped_plaintext)` ) — the SAME blob the Supabase path produces (so image/file/text all relay via the existing cloud crypto + `wrap_and_check_cloud_upload_plaintext`). Receiver: SyncManager.ingestRelaySseItem → cloud_decrypt(item_id, blob, syncKey) → storeItemWithLww (dedup on item_id → convergence with P2P/Supabase).

## Feature flag
`reqwest` is a workspace dep, currently daemon-gated behind `cloud-sync`. Add a NEW `relay-sync` feature pulling `reqwest` (independent of Supabase). No MSRV/pin impact (reuses workspace reqwest rustls-tls pin). Cloud crypto in copypaste-core is always-compiled.

## Security
Relay sees only ciphertext (XChaCha20 AAD item_id|schema). Token random (not derivable) → persist in keychain. Revocation on relay = sync-key/passphrase rotation (revoked device's pushes fail decrypt on readers); relay record self-evicts after 30d idle. Set RELAY_MAX_ITEM_BYTES + sync_ttl_secs to match AppConfig caps.
