# ADR-005: Slint as UI Framework

**Status:** Accepted
**Date:** 2026-05-23
**Deciders:** Project owner

## Context

CopyPaste needs a native desktop UI for macOS (and eventually Windows). The UI is
a long-running menu-bar app: it must boot fast, idle cheaply, render the history
list at 60fps while typing, and ship as a small bundle. The UI talks to the
daemon over a Unix socket (see ADR-002) and never links `copypaste-core`, so the
framework's only job is rendering and event handling — no business logic, no
storage, no networking.

The candidates evaluated were Rust-native UI toolkits:

1. **Slint** — declarative `.slint` DSL compiled to direct GPU/Skia rendering
2. **egui** — immediate-mode GUI, pure Rust
3. **iced** — Elm-architecture Rust GUI

## Decision

**Slint** is the chosen UI framework. All UI code lives in `crates/copypaste-ui/`
using Slint 1.8+.

## Rationale

| Criterion              | Slint                | egui                     | iced               |
|------------------------|----------------------|--------------------------|--------------------|
| Bundle size            | ~2 MB native         | ~5 MB                    | ~8 MB              |
| Startup time           | <100 ms              | <100 ms                  | ~200 ms            |
| Rendering model        | Retained, GPU/Skia   | Immediate, every frame   | Retained, custom   |
| macOS feel             | Native-ish           | Custom (basic)           | Custom             |
| Windows support        | Yes                  | Yes                      | Yes                |
| Rust integration       | Type-safe bindings   | Direct Rust              | Direct Rust        |
| Idle CPU (menu-bar)    | ~0% (retained)       | Higher (redraws/frame)   | ~0% (retained)     |
| Design quality ceiling | Medium (.slint DSL)  | Low (basic widgets)      | Medium             |
| License                | GPL / commercial     | MIT                      | MIT                |

**Why Slint over egui:**
- Immediate mode redraws every frame, which is wasteful for a menu-bar app that
  sits idle 99 % of the time (battery drain on laptops).
- Limited styling capabilities — would not pass the design bar we want even for
  beta.

**Why Slint over iced:**
- Larger bundle, slower cold start.
- Custom rendering looks further from native macOS than Slint's Skia output.

**Why Slint positively:**
- Tiny binary (~2 MB), fast cold start (<100 ms) — good fit for a menu-bar app.
- Retained-mode rendering: zero CPU when idle, smooth scrolling under load.
- `.slint` DSL is declarative and readable; layouts stay in markup, logic in Rust.
- Type-safe Rust bindings generated by `slint-build` at compile time — no
  runtime reflection, no string-keyed properties.
- Pure-Rust toolchain: no Node.js, no npm, no JS bundler, no WebView dependency.
  The whole UI builds with `cargo build` and ships as a single binary.
- One codebase for macOS and Windows; same `.slint` files render natively on both.
- Good Skia-based rendering quality on macOS — close to native widgets at the
  pixel level without needing a per-platform UI rewrite.

## Consequences

- All UI code in `crates/copypaste-ui/` using Slint 1.8+.
- No WebView, no JS toolchain, no Node.js/npm in the build pipeline.
- Windows support via the same Slint codebase — no second UI implementation.
- Design is constrained by Slint's styling system (no CSS); custom looks require
  custom Slint components.
- License: GPL for open-source use (acceptable for this project); a commercial
  Slint licence is available if licensing terms change later.

## Implementation

- `crates/copypaste-ui/ui/*.slint` — UI definitions (HistoryWindow, SettingsWindow, PairWindow).
- `crates/copypaste-ui/src/` — Rust bindings, IPC wiring, tray integration.
- Built via `slint-build` in `build.rs` — `.slint` → Rust at compile time.
